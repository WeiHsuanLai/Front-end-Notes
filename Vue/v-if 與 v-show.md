## 1. 底層原理的差異

### **v-if：真正的條件渲染 (Conditional Rendering)**

- **機制：** `v-if` 是「惰性」的。如果初始條件為假，它什麼都不會做。只有當條件第一次變為真時，它才會開始**編譯並掛載** DOM 節點。
    
- **切換時：** 當條件在 `true` 與 `false` 之間切換時，Vue 會完整地**銷毀（Destroy）**或**重建（Recreate）**該元素及其內部的子組件與事件監聽器。
    

### **v-show：單純的 CSS 切換**

- **機制：** 無論初始條件是什麼，元素**永遠都會被編譯並保留在 DOM 中**。
    
- **切換時：** 它只是簡單地切換 CSS 的屬性 `display: none;`。
    

---

## 2. 效能權衡 (Trade-offs)

這是在新工作中做決策時的參考標準：

| **特性**       | **v-if**                 | **v-show**           |
| ------------ | ------------------------ | -------------------- |
| **初始渲染開銷**   | **低** (如果不顯示就不渲染)        | **高** (不管顯示與否都會渲染)   |
| **切換開銷**     | **高** (需要頻繁銷毀/建立節點)      | **低** (只改 CSS)       |
| **DOM 狀態保留** | **否** (銷毀後狀態會消失)         | **是** (隱藏時狀態仍保留)     |
| **安全性/隱私**   | **高** (敏感資訊不渲染就不會出現在原始碼) | **低** (檢查原始碼看得到隱藏內容) |
|              |                          |                      |

## 3. 什麼時候該用哪一個？

### **優先使用 `v-if` 的場景：**

- **條件不常改變：** 例如根據使用者權限顯示「管理員面板」，一旦登入後通常不會頻繁切換。
    
- **大型組件：** 裡面包含大量子組件或複雜邏輯，如果使用者沒點開，就沒必要讓瀏覽器去負擔它的記憶體與計算。
    
- **確保生命週期執行：** 當你需要每次顯示時都重新觸發 `onMounted` 鉤子。
    

### **優先使用 `v-show` 的場景：**

- **頻繁切換：** 例如 Tab 標籤頁、折疊面板（Accordion）、或是滑鼠移入彈出的 Tooltip。
    
- **需要保留狀態：** 例如一個輸入框，即使暫時隱藏，使用者剛才打的字也不希望被清空。
    

---

## 4. 關鍵陷阱與建議

1. **搭配 `v-else`：** `v-if` 可以搭配 `v-else-if` 和 `v-else` 構建複雜邏輯，而 `v-show` 只能單獨使用。
    
2. **Vue 3 的優化：** 雖然 `v-show` 渲染開銷較高，但如果該元素內容非常簡單（例如只有一個 `<span>`），其實差異極小。
    
3. **不要與 `v-for` 並用：** 如同前面提到的，這會造成效能瓶頸。在 Vue 3 中，這甚至會因為優先級問題導致報錯。
    

**工程師的直覺小撇步：**

處理複雜的列表過濾顯示時，記住「切換頻繁用 `show`，路人甲乙用 `if`」這個原則就對了。
