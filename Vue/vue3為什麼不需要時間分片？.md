## 1. 什麼是時間分片？

在了解為什麼不需要之前，我們先看 React 為什麼要它。

當應用程式變得非常大、組件樹非常深時，JavaScript 的計算量會變得很恐怖。JS 是單執行緒的，如果它一口氣執行 200 毫秒，瀏覽器就會「卡死」，無法回應使用者的點擊或動畫（因為主執行緒被佔滿了）。

**React 的解法 (Fiber)：** 把任務拆成無數個小片段，每執行一小段就停下來，問瀏覽器：「有沒有更緊急的事（如點擊事件）？」如果有，就先讓給使用者，這就是時間分片。

## 2. Vue 3 為什麼不需要？（三個核心理由）

### A. 響應式系統的「精確導航」

這是最根本的原因。

- **React 的更新是「自頂向下」的：** 當狀態改變，React 預設會從根節點開始重新掃描整棵虛擬 DOM 樹（即使你用了 `memo` 也要進行比對）。這導致計算量與組件數量成正比。
    
- **Vue 的更新是「組件級別」的：** 透過強大的響應式系統，Vue 清楚地知道「是哪個組件」變了。當資料更新，Vue 只需要重新渲染那個受影響的組件及其子組件，而不需要掃描整棵樹。
    

> **結論：** 因為 Vue 的計算量本身就被控制在很小的範圍內，通常不會導致主執行緒卡死，自然不需要複雜的時間分片。

---

### B. 編譯器優化（Static Analysis）

Vue 3 的模板編譯器在打包時做了大量「手腳」，讓 Diff 過程變得極其輕量：

- **Patch Flags（動態標記）：** Vue 在編譯時會標記出哪些節點是動態的（例如只有文字會變）。Diff 的時候，Vue 會直接跳過所有靜態節點，只看動態的地方。
    
- **Hoisting（靜態提升）：** 靜態的 HTML 節點會被提到渲染函數之外，重複使用同一個記憶體位址，減少了垃圾回收（GC）的壓力。
    

這讓 Vue 3 在執行 Diff 時的速度遠快於 React，使得「長任務」發生的機率大幅降低。

---

### C. 複雜度與性價比的權衡

尤雨溪（Vue 的作者）曾明確表示過，引入時間分片是有代價的：

1. **架構複雜度：** 為了支持中斷與恢復，整個渲染架構必須變成非同步的（如 React Fiber），這會讓原始碼變得極難維護。
    
2. **效能損耗：** 任務拆分和排程本身也會帶來額外的開銷。
    
3. **開發者負擔：** 非同步渲染可能導致某些生命週期鉤子（Lifecycle Hooks）變得不可預測。
    

對於絕大多數 Web 應用程式來說，Vue 3 的「組件級更新 + 編譯優化」已經足夠流暢，不需要犧牲穩定性去換取一個 99% 情況下都用不到的「時間分片」。

---

## 3. 總結：兩者的取捨

|**特性**|**React (Fiber)**|**Vue 3**|
|---|---|---|
|**更新策略**|粗粒度（Pull），需要比對大範圍樹。|細/中粒度（Push），精確到組件。|
|**瓶頸解決**|透過**時間分片**讓出主執行緒。|透過**優化 Diff 效率**減少主執行緒佔用。|
|**渲染效能**|依賴排程機制。|依賴靜態編譯優化。|

**白話一點說：**

React 是因為家大業大（更新範圍大），所以需要「分期付款」（時間分片）才不會破產；而 Vue 3 則是精打細算（精確更新），每次更新的花費都很小，所以「現金支付」也完全沒問題。
