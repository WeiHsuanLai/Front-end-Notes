## 1. CSR (Client-Side Rendering) ── 用戶端渲染

這是現代單頁應用（SPA，如 React, Vue, Angular）最常見的模式。伺服器只回傳一個幾乎空白的 HTML 殼和一支龐大的 JavaScript 檔案。

- **運作流程：** 瀏覽器下載 JS $\rightarrow$ JS 執行 $\rightarrow$ 向 API 請求資料 $\rightarrow$ 渲染畫面。
    
- **優點：** 轉場流暢（不需重新整理）、減輕伺服器負擔。
    
- **缺點：** **FCP（首次內容繪製）慢**、SEO 極差（爬蟲可能只抓到空白頁）。
```
**SEO (Search Engine Optimization)**，中文稱為「**搜尋引擎最佳化**」。

簡單來說，它是一套透過**優化網站結構與內容**，來提升網站在搜尋引擎（如 Google、Bing）搜尋結果排名（Rank）的技術。當排位愈靠前，使用者點擊進入網站的機會就愈高，進而帶動流量。
```

---

## 2. SSR (Server-Side Rendering) ── 伺服器端渲染

傳統的網頁開發模式（如 PHP, JSP）或現代框架（如 Next.js, Nuxt.js）的進階運用。

- **運作流程：** 用戶請求 $\rightarrow$ 伺服器抓取資料並生成完整的 HTML $\rightarrow$ 瀏覽器直接顯示。
    
- **優點：** **SEO 極佳**、首屏加載快（使用者能立刻看到內容）。
    
- **缺點：** 伺服器壓力大、每次換頁都要重新請求（TTFB 較長）。

```
TTFB (Time to First Byte)  是衡量網頁載入效能的關鍵指標之一，中文常譯為「**首位元組時間**」。

簡單來說，它代表從你**發出請求**（點擊連結或輸入網址）到瀏覽器**接收到伺服器回傳的第一個位元組資料**之間所經過的時間。
```

---

## 3. SSG (Static Site Generation) ── 靜態網站生成

在**建置階段（Build Time）**就將所有頁面預先產生成 HTML 檔案。

- **運作流程：** 開發者編譯代碼 $\rightarrow$ 產生所有靜態 HTML $\rightarrow$ 部署到 CDN。
    
- **優點：** 速度極快（接近極限）、安全性高、主機成本低。
    
- **缺點：** 資料更新需重新建置、不適合具有大量動態資料（如電商庫存）的網站。
    

---

## 4. ISR (Incremental Static Regeneration) ── 增量靜態再生

這是 SSG 的進化版，由 Next.js 普及，解決了靜態頁面無法即時更新的問題。

- **核心機制：** 允許你在背景更新已生成的靜態頁面，而無需重新部署整個網站。你可以設定一個 `revalidate` 時間（例如 60 秒），時間到後若有新請求，伺服器會默默更新該頁面。
    

---

## 5. Hydration（注水 / 活化）

這不是一種渲染模式，而是 CSR 與 SSR/SSG 結合的關鍵過程。

當伺服器送出靜態 HTML 後，瀏覽器會加載 JS，並將事件監聽器（Event Listeners）掛載到現有的 HTML 節點上，讓網頁從「純看」變成「可互動」。

> **注意：** 如果 JS 太大，會出現「看得見卻點不動」的 **TTI (Time to Interactive)** 延遲。

| **模式**  | **內容生成時機** | **SEO 友善度** | **伺服器負擔** | **使用場景**          |
| ------- | ---------- | ----------- | --------- | ----------------- |
| **CSR** | 瀏覽器執行時     | 低           | 極低        | 後台管理系統、需頻繁互動的 App |
| **SSR** | 用戶請求時      | 高           | 高         | 需即時資訊的電商、新聞入口網    |
| **SSG** | 專案建置時      | 高           | 無 (CDN)   | 部落格、說明文件、形象官網     |
| **ISR** | 背景定時更新     | 高           | 中         |                   |
